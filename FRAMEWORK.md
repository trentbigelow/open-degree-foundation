## 1. Framework Philosophy

### Why This Exists

The traditional undergraduate degree measures **time served at an institution**. The Open Degree Framework measures **what you know and can demonstrate**.

These are not the same thing. A framework built around demonstrated competency is:

- **More honest** — evidence beats brand
- **More equitable** — accessible regardless of financial circumstance or life path
- **More useful to employers** — specific, verified, and current
- **More respectful of prior learning** — work, life, and self-directed study count

### Design Principles

**Evidence is the credential.**
Every competency must be verified with documented, linkable, checkable evidence. Enrollment is not evidence. Tuition paid is not evidence. The work product is the evidence.

**Transparency over opacity.**
The full framework specification is public. Employers, learners, and contributors can read exactly what each competency requires. No black box.

**Prior learning has value.**
What you learned founding a company, leading a team, or through self-directed study is real knowledge. A framework that ignores this is not measuring learning — it is measuring access to institutions.

**Open source or it doesn't scale.**
No single organization should own the standard for how humans represent their learning. The ODF is CC BY 4.0. Fork it, adapt it, build your own track.

**Rigor is non-negotiable.**
This framework is not a shortcut. The evidence standards are strict. Capstones are independently peer-reviewed. Prior learning claims require real documentation. The goal is more signal, not less accountability.

## 2. Degree Structure

A complete ODF credential requires:

| Component | Requirement |
|---|---|
| **Core Foundations** | All 6 domains — required for every track |
| **Specialization Track** | All required domains + electives for chosen track |
| **Capstone Project** | 1 integrative, public, peer-reviewed project |
| **Comprehensive Learner Record** | Assembled CLR with all credentials linked |

### Credit Hour Equivalence

| Component | Equivalent Credit Hours |
|---|---|
| Core Foundations (6 domains) | ~30 credits |
| Specialization Track | ~60 credits |
| Electives (within track) | ~12 credits |
| Capstone | ~6 credits |
| General prior learning bridge | ~12 credits |
| **Total** | **~120 credits** |

This maps to standard US undergraduate degree requirements. The ODF does not award credit hours — it awards **verified competency credentials** — but the equivalence is documented for reference.

### Time to Completion

| Learner Profile | Estimated Time |
|---|---|
| Recent graduate with ~60 prior credits | 12–18 months |
| Career professional with extensive PLA | 6–12 months |
| New learner with no prior college | 2–3 years |
| Full-time study (20+ hrs/week) | Reduce estimates by ~40% |

## 3. Verification Tiers

Every competency must be verified at **Tier 1, 2, or 3**. Tier 1 is preferred. All tiers produce verifiable credentials.

---

### Tier 1 — Standardized External Exam ⭐⭐⭐

*Highest credibility. Independent, third-party proctored. Scores are objective and universally interpretable.*

**Accepted exam types:**

| Exam Program | Coverage | Cost | Notes |
|---|---|---|---|
| **CLEP** (College Level Examination Program) | 33 subjects across core academic domains | ~$90/exam | Accepted at 2,900+ accredited colleges |
| **DSST** (DANTES Subject Standardized Tests) | 37 subjects, more professional/applied | ~$85/exam | Originally for military, now public |
| **AP Exams** | High school advanced subjects | ~$98/exam | Valid if score ≥3; older scores accepted |
| **Professional Certifications** | Domain-specific (AWS, PMI, CFA, etc.) | Varies | Accepted where mapped in track spec |
| **GRE Subject Tests** | Advanced academic domains | ~$160/exam | For upper-division equivalency |

**Minimum passing standards:**
- CLEP: Score of 50 or higher (equivalent to C or better in college course)
- DSST: Score of 400 or higher
- AP: Score of 3 or higher
- Professional certifications: Pass (vendor-defined)

---

### Tier 2 — Platform-Verified Course Completion ⭐⭐

*Identity-verified enrollment with proctored or authenticated assessments from recognized institutions.*

**Accepted platforms:**

| Platform | Requirements for Tier 2 |
|---|---|
| **Coursera** | Verified Certificate (identity + assignment verification) |
| **edX** | Verified Certificate |
| **MIT Open Learning Library** | Interactive course with auto-graded assessments |
| **Udacity** | Nanodegree with project review |
| **LinkedIn Learning** | Certificate + associated skill assessment |
| **DeepLearning.AI** | Course certificate with graded assignments |
| **fast.ai** | Course completion with project portfolio |

**Requirements for Tier 2 acceptance:**
- Certificate must be issued under verified identity (not audit track)
- Course must include graded assessments, not just video completion
- Certificate URL must be publicly accessible and logged in the CLR
- Institution offering the course must be accredited or have demonstrated academic rigor

**Not accepted as Tier 2:**
- Audit completions with no graded work
- Self-reported completion without certificate
- Certificates from platforms with no assessment component

---

### Tier 3 — Portfolio & Prior Learning Documentation ⭐

*Demonstrated learning through prior formal education, professional work, published output, or peer-reviewed project.*

**Accepted evidence types:**

| Evidence Type | Requirements |
|---|---|
| **Prior college transcripts** | Official transcript from regionally accredited institution; course must map to competency |
| **Professional work products** | Documented artifacts (reports, strategies, code, designs) with dates and context |
| **Published writing** | Articles, white papers, books — publicly accessible, author verified |
| **Shipped products** | Documented role, scope, and outcomes; product must be verifiable |
| **Employment record** | LinkedIn + supporting documentation; role must directly exercise competency |
| **Startup/venture record** | Founding documentation, Crunchbase/public record, investor letters |
| **Peer-reviewed capstone** | Project evaluated by 2+ independent ODF reviewers |
| **Recorded oral examination** | 30–60 min recorded discussion with 2+ reviewers; rubric-based scoring |

**Requirements for Tier 3 acceptance:**
- Evidence must be specific, dated, and verifiable by a third party
- Self-assertion without supporting documentation is not accepted
- Each Tier 3 claim requires a written reflection (300–500 words) connecting the evidence to the competency standard
- Claims are logged in the CLR with links to evidence

**Tier 3 limits:**
- Up to 60% of total competencies may be satisfied via Tier 3
- The remaining 40% must include at least Tier 2 verification
- No single competency may be claimed at Tier 3 without supporting documentation

##4. Prior Learning Assessment (PLA)

### What Counts as Prior Learning

Prior learning is any knowledge or skill acquired outside of the courses taken to complete this framework. It includes:

- College coursework at any accredited institution (any year, no expiration)
- Professional experience and career history
- Entrepreneurship, founding, and business leadership
- Self-directed study with demonstrable outputs
- Military training and service
- Professional certifications and licenses
- Published research, writing, or creative work
- Open source contributions

### How to Claim PLA Credit

1. **Identify the competency** you are claiming prior learning for
2. **Gather evidence** (transcript, work product, employment record, published work)
3. **Write a reflection** (300–500 words) explaining how the evidence maps to the competency standard
4. **Submit to the PLA log** in your degree plan file (`my_degree_plan.md`)
5. **Link all evidence** — every claim must have a checkable link or attached document
6. **Flag for peer review** if the claim is complex or if you want external validation

### PLA Decision Standards

| Claim Strength | Description | Decision |
|---|---|---|
| **Strong** | Direct, documented, dated evidence clearly matching competency | Auto-accepted; logged |
| **Moderate** | Related evidence requiring interpretation | Accepted with written reflection |
| **Weak** | Indirect or undocumented claim | Requires Tier 2 coursework to supplement |
| **Invalid** | Self-assertion only, no supporting documentation | Rejected; coursework required |

### PLA Limits

| Rule | Limit |
|---|---|
| Maximum PLA-covered competencies | 60% of total |
| Minimum Tier 1 or Tier 2 competencies | 40% of total |
| Prior college credits | No expiration — all years accepted |
| Professional experience | Must be documented and verifiable |

### 5.1 Quantitative Reasoning & Data Literacy

**Competency Standard**

The learner can interpret statistical findings, apply basic probability concepts, reason quantitatively about real-world problems, evaluate data-based claims critically, and communicate quantitative information clearly to non-technical audiences.

**What "competent" looks like:**
- Reads and interprets a statistical result (mean, median, confidence interval, p-value) correctly
- Designs or evaluates a simple experiment or A/B test
- Identifies common quantitative fallacies (base rate neglect, Simpson's paradox, etc.)
- Translates a data finding into a clear, non-technical recommendation

**Suggested Courses**

| Course | Institution | Platform | Hours | Cost |
|---|---|---|---|---|
| Introduction to Statistics | Stanford | Coursera | 15h | Free/Verify |
| Statistics & Probability | Khan Academy | Khan Academy | 40h | Free |
| Data Science: Statistics & ML | Johns Hopkins | Coursera | 30h | Free/Verify |
| Mathematics for Machine Learning | Imperial College London | Coursera | 48h | Free/Verify |
| Intro to Statistics | Udacity | Udacity | 15h | Free |

**Assessment Options**

| Tier | Assessment | Minimum Standard |
|---|---|---|
| Tier 1 | CLEP College Mathematics | Score ≥ 50 |
| Tier 1 | CLEP Introductory Statistics | Score ≥ 50 |
| Tier 2 | Verified certificate from any course above | Pass with graded assessment |
| Tier 3 | USC/prior transcript: statistics or calculus course | Grade C or better |
| Tier 3 | Professional portfolio: documented data analysis work | 2+ examples with reflection |

The learner can write clearly and persuasively, construct evidence-based arguments, communicate complex ideas to varied audiences (technical and non-technical), and edit their own work for clarity, precision, and logical flow.

**What "competent" looks like:**
- Writes a clear, structured argument with a thesis, evidence, and conclusion
- Adapts tone and complexity for different audiences
- Edits for concision — removes filler, tightens prose

**What "competent" looks like:**
- Identifies the logical form of an argument and evaluates its validity
- Spots common fallacies (straw man, ad hominem, false dichotomy, etc.) in real-world text
- Applies inductive reasoning to evaluate empirical claims

| Education | `/tracks/education/track_spec.md` | 🔜 Planned |

### Track Specification Requirements

All track spec files must include:

1. **Track overview** — who it's for, what it prepares learners to do
2. **Required competency domains** — minimum 6, maximum 12
3. **Elective options** — minimum 2 choices required from defined list
4. **Competency standards** — clear definition of "competent" for each domain
5. **Suggested courses** — minimum 3 per domain, from accessible platforms
6. **Assessment options** — Tier 1, 2, and 3 options for each domain
7. **PLA examples** — what prior professional experience counts
8. **Track maintainer** — GitHub handle responsible for keeping it current

### Proposing a New Track

1. Open a GitHub issue with the label `new-track`
2. Include: track name, target audience, draft competency list, your qualifications to maintain it
3. Two maintainer approvals required to begin drafting
4. Draft track spec submitted as pull request
5. Community review period: 30 days
6. Merged on 2 maintainer approvals with no blocking objections

## 7. Capstone Project

The capstone is a required, integrative, public project that demonstrates mastery across multiple competency domains.

### Requirements

| Requirement | Standard |
|---|---|
| **Public** | Published openly and accessible by URL |
| **Substantial** | Minimum 20 hours of documented effort |
| **Integrative** | Draws on at least 3 competency domains (Core + Specialization) |
| **Peer reviewed** | Evaluated by at least 2 independent ODF community reviewers |
| **Self-documented** | Written reflection (500–1,000 words) on what was learned and how it connects to the framework |

### Accepted Capstone Types

| Type | Description | Public Output |
|---|---|---|
| **Original Research** | Research question, methodology, analysis, findings | Published paper or report |
| **Product Launch** | Designed, built, and shipped a real product with documented outcomes | Public product + case study |
| **Strategy Document** | Full business, product, or policy strategy for a real organization | Published white paper |
| **Technical Implementation** | AI or software system designed and built | Public repo + architecture writeup |
| **Portfolio Defense** | Oral examination reviewing work across all domains | Recorded video + transcript |
| **Framework or Curriculum** | An open-source educational or professional framework | Published repo |

### Peer Review Process

1. Learner submits capstone to the `/community/capstone-review` issue queue
2. Two reviewers from the ODF community volunteer (or are assigned by maintainers)
3. Reviewers evaluate against the competency domains claimed and the capstone rubric
4. Rubric scores and comments submitted within 30 days
5. Learner may revise and resubmit once if initial review is not passing
6. Passing capstone is logged in the CLR with reviewer names and date

### Capstone Rubric

| Dimension | 1 — Insufficient | 2 — Developing | 3 — Competent | 4 — Exemplary |
|---|---|---|---|---|
| **Competency integration** | Single domain only | 2 domains referenced | 3+ domains clearly applied | Sophisticated multi-domain synthesis |
| **Evidence of learning** | Assertion only | Some documentation | Clear evidence trail | Comprehensive, detailed documentation |
| **Public accessibility** | Not public | Partially accessible | Fully accessible | Well-organized and navigable |
| **Written reflection** | Absent or superficial | Describes what was done | Connects work to framework | Insightful, self-critical, forward-looking |
| **Rigor** | Low effort, incomplete | Moderate effort | Substantial effort and quality | Exceptional quality |

**Passing standard:** Average score of 3.0 or higher across all dimensions, with no dimension below 2.

## 8. Comprehensive Learner Record (CLR)

### What It Is

The Comprehensive Learner Record is the learner's full, assembled, verifiable credential portfolio. It is:

- **Learner-owned** — stored in the learner's digital wallet, not on ODF servers
- **Cryptographically signed** — each component badge is signed by the ODF issuing key
- **Employer-readable** — follows W3C Verifiable Credentials and CLR 2.0 standards
- **Portable** — shareable to any employer, institution, or platform that reads open standards

### CLR Components

| Component | Standard | Description |
|---|---|---|
| **Per-competency Open Badges** | Open Badges 3.0 | One signed badge per completed competency |
| **Capstone credential** | Open Badges 3.0 | Signed badge for completed, peer-reviewed capstone |
| **Full CLR portfolio** | CLR 2.0 (1EdTech) | Complete assembled record with all badges, evidence links, and metadata |
| **Learner narrative** | Free text | Self-authored statement (500 words) contextualizing the full record |

### What Each Open Badge Contains

```
Badge: [Competency Name]
Issuer: Open Degree Foundation
Earner: [Learner Name]
Issued: [Date]
Criteria: [Link to FRAMEWORK.md competency standard]
Evidence: [Link to certificate, transcript, or portfolio artifact]
Assessment Tier: [1 / 2 / 3]
Cryptographic Proof: [W3C VC signature]
Verify: odf.github.io/verify/[badge-hash]
```

### Verification

Any badge or CLR can be verified at:
```
odf.github.io/verify/[credential-hash]
```

Verification confirms:
- Credential is authentic (not fabricated)
- Issuer is the Open Degree Foundation
- Earner matches the credential holder
- Date of issue is accurate
- All evidence links are live and accessible

### CLR Generation

CLR generation tooling is under development in `/tools/`. Until tooling is available, learners assemble their CLR manually using the template in `/templates/clr_template.md` and submit for badge issuance via the issue queue in `/community/`.

## 9. Governance & Appeals

### Maintainers

The framework is maintained by the community via GitHub. Current maintainers are listed in `MAINTAINERS.md`. Maintainers are responsible for:

- Reviewing and merging pull requests to the framework spec
- Approving new specialization tracks
- Overseeing the appeals process
- Updating course recommendations as the landscape evolves

### Curriculum Changes

Any change to a competency standard, verification requirement, or assessment option requires:
1. A GitHub issue proposing the change with rationale
2. 30-day community comment period
3. 2 maintainer approvals
4. No blocking objections from active community members

### PLA Appeals

If a prior learning claim is rejected and the learner believes the decision is incorrect:
1. Submit an appeal via GitHub issue with label `pla-appeal`
2. Provide additional evidence or context
3. A 3-person panel (2 maintainers + 1 community member) reviews within 30 days
4. Panel decision is final

### Capstone Appeals

If a capstone receives a failing review and the learner believes the review was conducted incorrectly (not simply that they disagree with the score):
1. Submit an appeal via GitHub issue with label `capstone-appeal`
2. A different set of 2 reviewers conducts a fresh review
3. The higher of the two scores stands

## 10. Versioning

The ODF framework follows semantic versioning: `MAJOR.MINOR.PATCH`

| Change Type | Version Bump | Example |
|---|---|---|
| Breaking change to competency standards | MAJOR | Removing a core domain |
| New feature, backward compatible | MINOR | Adding a new assessment option |
| Corrections, clarifications | PATCH | Fixing a broken course link |

All learners' credentials reference the framework version active at the time of their completion. Version history is maintained in `CHANGELOG.md`.

**Current version:** `0.1.0-draft`
**Stable release target:** `1.0.0` — Q3 2026
